Introduction

Document Purpose

This Software Requirements Specification (SRS) describes, in a structured and moderately engineered manner, the requirements for the AI-Generated 3D House Customization Web App. The document formalizes functional and non-functional requirements, interface descriptions, system behavior, constraints, assumptions, and the data vocabulary used by the system.

The SRS serves as a contractual and traceable artifact between stakeholders (supervisors, evaluators, developers, testers, and mildly curious onlookers) and the development team. It will guide design, implementation, testing, and evaluation of the system, and can be used as a reference during FYP demos and viva sessions.

Product Scope

The product is a web-based application that enables users to:

• Generate AI-created, modular 3D house structures (voxel-based / low-poly).

• Customize these structures interactively using a drag-and-drop interface for modular components (walls, windows, doors, and other paraphernalia).

• Navigate and inspect the 3D house using intuitive camera controls (orbit, zoom, pan).

• Optionally save and reload custom designs for demonstration or iterative tweaking.

Primary goals:

• Provide a visually compelling FYP demo without requiring users to know Blender, Maya, or the arcane secrets of 3D modeling.

• Showcase integration of AI (generative models) with real-time 3D rendering in the browser.

• Maintain lightweight performance suitable for mid-range machines used in university labs.

Overall Description

Product Perspective

The AI-Generated 3D House Customization Web Application is a standalone, web-based system designed to demonstrate the integration of artificial intelligence with interactive 3D visualization technologies. It is not derived from or dependent on any previous version of similar software; instead, it introduces a novel workflow for generating modular architectural structures using AI models and rendering them directly within a browser environment. The system functions independently but may optionally connect to a backend service for AI inference and design persistence, depending on deployment configuration.

The application acts as an intermediary between the user, an AI content generator, and a real-time 3D rendering engine. The user interacts through a graphical interface built on web technologies, which sends generation requests to the AI module based on selected parameters, such as floor count. The AI module returns a lightweight modular house structure that can be customized using standardized components like doors and windows. These elements are manipulated in a 3D viewport powered by WebGL-based frameworks. Optional communication with a backend database may be used to save and retrieve design states for future modification.

Product Functionality

High-level functions include:

AI House Generation

  • Generate a modular 3D house for a given floor count.
  • Enforce low-poly, lightweight geometry.

Floor Plan Configuration

  • Single-floor and two-floor selector.
  • Visual indication of selected floor mode.

Component Customization

  • Drag-and-drop placement of windows/doors.
  • Snap-to-wall and snap-to-grid behavior.
  • Validation to prevent invalid placements or overlaps.

3D Visualization and Navigation

  • Orbit, pan, zoom controls.
  • Predefined camera views (e.g., front, side, isometric).
  • Optional Ghost Mode / Zen Mode.

Persistence (Optional)

  • Save current design to local storage or backend.
  • Load previously saved designs.
  • Warning before overwriting existing designs.

System Feedback

  • Loading indicators for AI generation.
  • Error and success notifications.
  • Rare but delightful micro-interactions (e.g., tiny bouncing house icon while generating).

Operating Environment

The system runs primarily in a web environment. On the client side, it requires a modern desktop browser (Chrome, Firefox, or Edge, recent versions) with WebGL support and a GPU capable of basic 3D rendering. A minimum display resolution of 1366×768 is recommended. The server side (if enabled) consists of a lightweight backend such as Node.js/Express or Python/Flask, which exposes the AI model through a REST API. If heavy inference is required, the server may utilize a GPU; otherwise, the system can rely on cached or lightweight models. A stable internet connection is necessary whenever server-based AI or data persistence is used. The system must coexist with standard OS/browser environments and does not require any special local installations.

Minimum Platform Requirements:

  • Client: Modern browser with WebGL, GPU capable of basic 3D acceleration, ≥1366×768 resolution.
  • Server (optional): Node.js or Python runtime, REST API endpoint, optional GPU.
  • Network: Stable connection for API-based inference.

Hardware Interfaces

The system interacts with standard client hardware and an optional server-side machine used for AI inference. These interfaces define how input devices, GPUs, and networking components are utilized by the application.

  • 1. Client Device Interface

The client interacts with the system through standard input and graphics hardware.

  • Input Devices: Mouse or trackpad for navigation and object manipulation; keyboard may be used for optional shortcuts.
  • Graphics Hardware: A GPU capable of running WebGL, enabling real-time 3D rendering. The system communicates with the GPU indirectly through the browser using WebGL libraries such as Three.js or Babylon.js.
  • Data/Control Interaction: Input events (clicks, drags, keypresses) are captured through browser event listeners, while rendering commands are issued to the GPU via WebGL.
  • 2. Server Machine Interface (Optional)

If server-side AI inference is used, the system interacts with a remote machine that provides GPU-accelerated computation.

  • Compute Hardware: A server GPU used to run AI models.
  • Network Interface: Communication occurs over standard HTTP/HTTPS requests to a REST API endpoint.
  • Data/Control Interaction: The client sends text or structural prompts to the API, and the server returns generated model data or parameters. Communication is handled through tools such as the Fetch API, Axios, or backend frameworks like Express or Flask.

Software Interfaces

The system operates entirely within a web browser environment, so its interaction with the operating system is indirect and mediated through standard browser APIs. The software is expected to run on Windows, macOS, and mainstream Linux distributions, provided they support a compatible modern browser. All OS-level file access, graphics handling, and hardware communication are managed by the browser rather than the application itself.

The application relies on WebGL for GPU-accelerated rendering, which the browser maps to the operating system’s graphics drivers. Any local data storage uses browser-provided mechanisms such as Local Storage or IndexedDB; no direct file system access is performed. Input events (mouse, trackpad, keyboard) are delivered through the OS → browser → web application pipeline, requiring no OS-specific handling within the system. Network communication, when used, is routed through the OS's standard networking stack but accessed using browser APIs like Fetch. The system therefore requires no OS-level dependencies beyond a functioning browser environment and compatible graphics drivers.

Communications Interfaces

The system communicates with an optional backend service using standard web protocols. All client–server interactions occur over HTTPS to ensure encrypted data transmission. Requests and responses use lightweight JSON formatting for sending prompts, receiving AI-generated structural data, and exchanging any saved design information. No specialized messaging formats or nonstandard protocols are required, as communication relies on typical browser networking capabilities such as the Fetch API.

Security considerations include encrypting all data in transit and minimizing any personally identifiable information exchanged with the server. Only design-related data and model parameters are transmitted. If implemented, the backend follows standard web security practices and may use common authentication measures, though the system does not require high-risk or high-volume data transfer. Overall, communication remains simple, using widely adopted web standards to maintain compatibility and reliability.

Other Non-functional Requirements

Performance Requirements

  • 3D House Generation Time
    The AI-generated house model must fully load and appear in the 3D canvas within 2 minutes on a moderate-performance machine (e.g., 4-core CPU, integrated GPU, 8–16 GB RAM).
    Rationale: Ensures a responsive first-load experience and supports iterative design workflows.
  • Drag-and-Drop Interaction Latency
    Drag-and-drop actions (e.g., moving furniture, resizing components) must have a response time of under 100 ms between user movement and visible 3D update.
    Rationale: Keeps interactions feeling natural and prevents motion–input desynchronization.
  • Rendering Frame Rate
    The 3D environment should maintain a minimum of 30 frames per second (FPS) on modern browsers running on integrated GPUs.
    Rationale: Ensures smooth visualization and prevents motion sickness or usability issues.
  • Floor Toggle Re-Rendering Time
    Toggling between floors (e.g., ground floor to second floor) must trigger regeneration and a completed re-render within 3 seconds.
    Rationale: Fast navigation between floors supports fluid design and review.
  • Camera Motion Responsiveness
    Camera orbit, pan, and zoom interactions must update visually in real time with no noticeable lag, maintaining at least 30 FPS during movement on supported devices.
    Rationale: Stable and smooth navigation is critical for inspecting 3D structures.

Safety and Security Requirements

Safety Requirements

  1. Valid Component Placement
    The system shall prevent placement of components in impossible or invalid positions (e.g., floating doors, unsupported walls).
  2. Collision/Overlap Prevention
    The system shall detect and block components from overlapping in the same physical space.
  3. Destructive Action Warnings
    The system shall warn users and request confirmation before destructive actions such as overwriting designs or resetting layouts.

Security Requirements

Expected Security Level:
Medium-level security suitable for an academic/prototype environment, following standard web protection practices.

Major Requirements:

  1. Validate and sanitize all data sent to backend APIs.
  2. Restrict backend endpoints to authenticated/authorized users (e.g., token or IP-based access).
  3. Use HTTPS for all client–server communication.
  4. Avoid storing sensitive data; design files should contain only structural information.
  5. Ensure users can access only their own saved designs.

Software Quality Attributes

4.3.1 Usability

  • UI shall be visually clean with clearly labeled controls and tooltips.
  • The core workflow (Generate → Customize → View) must be discoverable without reading documentation.
  • Error messages must be clear, human-readable, and free of technical stack traces.
  • Priority: Ease of use over ease of learning, ensuring smooth interaction for new users.

4.3.2 Maintainability

  • Use a modular React architecture, separating UI/layout components, 3D rendering modules, and AI/data service layers.
  • Apply consistent naming conventions and comment critical logic (snapping, collision detection, floor switching).
  • AI model integration must be swappable with minimal changes to other modules (design for change).

4.3.3 Reliability

  • Snap-to-grid and snap-to-wall logic shall reduce user placement errors and prevent edge-case failures.
  • If critical services fail (e.g., AI offline), the system must show a clear error instead of a blank view or crash.

4.3.4 Portability

  • The application shall run on all major WebGL-capable browsers with no plug-ins required.
  • Use glTF as the standard 3D format to ensure compatibility across rendering engines and future extensions.
  • UI must adapt to different screen resolutions (desktop/laptop first, tablet optional).

4.3.5 Extensibility

  • Architecture must support adding new component types (balconies, fences, trees, etc.) with minimal refactoring.
  • Internal feature flags (e.g., ghostModeEnabled, zenModeEnabled) shall allow quick implementation of experimental visual modes.
  • The data schema must be versioned to avoid breaking old designs when new attributes are added.

4.3.6 Testability

  • Core logic (snapping, placement validation, state updates) shall be structured for unit testing.
  • Rendering and AI calls should be abstracted behind interfaces to allow mocking during automated tests.
  • Key workflows (generate, save, load, customize) must have repeatable test cases.

4.3.7 Interoperability

  • 3D assets, saved designs, and metadata must follow open standards (glTF, JSON) for compatibility with external tools.
  • The API layer should use REST-style endpoints to allow integration with future services (e.g., furniture libraries, material selectors).

Other Requirements

Future scope includes:

  • Enhanced textures and materials (brick, wood, glass).
  • Interior layout generation (rooms, furniture).
  • More complex house styles (modern, traditional, slightly haunted).
  • VR integration using WebXR for immersive viewing.

Potential gamification elements:

  • Non-intrusive design scores (e.g., natural light rating, aesthetic score).
  • Optional chaosModeEnabled flag for internal experimentation, not exposed to normal users.
